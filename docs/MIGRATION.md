# Migrating from the YAML package to the integration

The HACS integration is a **rewrite** of `packages/mxz_coordinator.yaml`. The behavior
is identical (same thresholds, same decide→act→self-heal logic), but the integration
**owns its own entities** instead of relying on `input_*` helpers, so the entity IDs
change. Automations or dashboards that reference the old IDs must be updated.

## Entity ID mapping

| YAML package (`input_*` helper)        | Integration entity                  |
| -------------------------------------- | ----------------------------------- |
| `input_number.hvac_primary_target`     | `number.*_primary_target`           |
| `input_number.hvac_secondary_target`   | `number.*_secondary_target`         |
| `input_boolean.hvac_primary_enable`    | `switch.*_primary_enable`           |
| `input_boolean.hvac_secondary_enable`  | `switch.*_secondary_enable`         |
| `input_boolean.hvac_coordinator_enable`| `switch.*_coordinator_enable`       |
| `input_boolean.hvac_eco_idle`          | `switch.*_eco_idle`                 |
| `input_select.hvac_shared_mode`        | `select.*_shared_mode`              |
| `input_datetime.hvac_last_mode_change` | *(internal — no entity)*            |
| `sensor.mxz_plan`                       | `sensor.*_plan`                     |

The `*` prefix is generated by Home Assistant from the integration's device name
(e.g. `number.mxz_coordinator_primary_target`). The plan sensor exposes the same
decision attributes as before (`primary_demand`, `secondary_engage`, `standoff`, …),
plus `primary_fan_hold` / `secondary_fan_hold` (who's holding the fan — added v2.15.0).

Since **v2.1.0** the integration also creates two native single-target thermostats
(`climate.*_primary_thermostat` / `climate.*_secondary_thermostat`) — see
"Single-target HomeKit/Google tile" below.

## Upgrade steps

1. **Note your current setup** — screenshot your target values, which rooms are
   enabled, and any automations that read/write the old helpers.
2. **Install the integration** via HACS (see the README) and run the config flow:
   pick your two heads and two temperature sensors.
3. **Remove the YAML package** (or comment out its `!include`) so the old `input_*`
   helpers and automations stop fighting the integration over the same heads.
4. **Re-point automations/dashboards** at the new entity IDs (table above).
5. **Turn on** `switch.*_coordinator_enable`, set the targets, and enable the rooms.
   Tunable constants (S, D, hysteresis, eco extremes, clamp, resting-mode bias) live in
   the integration's **Configure** → options dialog.

## Rollback

The legacy package is still in `packages/mxz_coordinator.yaml`. To revert: delete the
integration from **Settings → Devices & Services**, re-add the package `!include`, and
reload YAML.

## Single-target HomeKit/Google tile (v2.1.0+)

The integration now ships a **native single-target thermostat per room**:

- `climate.*_primary_thermostat`
- `climate.*_secondary_thermostat`

Each is a clean one-number "Auto" tile (`off` / `heat_cool`, a single setpoint — never a
dual heat/cool threshold) that binds directly to HomeKit/Google. It's a thin facade over
the room's `number.*_target` and `switch.*_enable` entities: setting the temperature drives
the target, turning it off disables the room, and the coordinator stays the sole writer to
your real heads. **Expose only these two `climate.*` thermostats** to HomeKit/Google — not
the raw head `climate` entities (two tiles per room would fight over the same heads).

Optional **vane control**: if your heads expose vertical/horizontal vane `select` entities,
pick them in the config flow and they appear as swing modes on the thermostat tile.

## Resting-mode bias (v2.2.0+)

When **no room is calling**, the coordinator has to pick a shared mode to idle in. By default
it holds whatever was **last called** — but that can leave the system sitting in the wrong mode
for the season (e.g. resting in `heat` on a summer day until a real cool demand finally arrives).

The **Resting mode** option (Configure → options) lets you bias the idle mode:

| Setting | Idle behavior |
| --- | --- |
| `last` (default) | Hold the last called mode. Unchanged from earlier versions. |
| `cool` | Always settle on `cool` when no room is calling. |
| `heat` | Always settle on `heat` when no room is calling. |

The bias changes **only the neutral resting mode** — a genuine opposite demand (a room past its
demand threshold) still flips the shared mode, and the flip obeys the same mode hysteresis as any
other change. The default `last` preserves existing behavior, so upgrading needs no action.

**YAML package:** the same control is `input_select.hvac_resting_bias` (options `last` / `cool` /
`heat`). It has no `initial:`, so your choice persists across restarts; an unset value reads as `last`.

### No longer need the echavet proxy for the tile

Earlier versions relied on the echavet `mitsubishi_climate_proxy`
`coordinator_single_target` mode for the single-target surface. That mode is hardcoded to
write the legacy package's `input_*` helpers, so it pairs with the YAML package, **not** this
integration's entities. With the native thermostats you don't need the proxy at all. (The
`mxz_recompute` event is still honored, so any existing proxy/automation nudge keeps working.)

## Re-engage drift & engage latch (v2.14.0+)

A room now runs all the way to its target and then coasts. The **re-engage drift**
(Configure → options) sets how far it may wander back before conditioning resumes — it
replaces the old fixed engage deadband as the re-engagement gate. No new entities; the
default reproduces the earlier feel. This release also dropped the short-lived
coast-offset option — if you ever saved one, it's simply ignored.

## Fan boost & the Fan auto switch (v2.10.0+ / v2.15.0+ / v2.16.0+)

**Fan boost defaults ON from v2.10.0.** If you never saved the option, the
delta-proportional fan is now active; opt out anytime under **Configure → Fan boost**.
An explicitly-saved off stays off.

**Manual fan latch & max-speed handback (v2.15.0).** Pick a fan speed yourself on any
head and I stop writing that head's fan until you hand it back — set `auto`, or (from a
slider with no `auto` stop) slide to max while boost would already be commanding max. A
hold at any slower speed is a ceiling you chose: it never releases on its own. No new
entities; the plan sensor gains `primary_fan_hold` / `secondary_fan_hold` (bool) so a
dashboard can show who's driving the fan. Purely additive.

**v2.16.0 adds a per-room Fan auto switch** — two new entities on the existing device:

- `switch.*_primary_fan_auto`
- `switch.*_secondary_fan_auto`

Each is a live mirror of that room's manual-fan hold (ON = boost drives, OFF = a manual
speed is held) and the discoverable way to hand fan control back — see the README's
"Who drives the fan". Purely additive: no existing entity IDs change, and the switches
store no state of their own (they mirror the coordinator's latch, which seeds from
observed head state on restart).

## Prompt refresh on fan changes (v2.16.1+)

I now recompute promptly when a head's `fan_mode` changes, so a manual hold — or a
handback — shows up on the tile and the Fan auto switch right away instead of waiting for
the next heartbeat. Behavior only: no entity or option changes.

## Live airflow display (v2.17.0+)

Optional per-room **airflow sensor** (`primary_stage` / `secondary_stage`), auto-detected
at setup from the head's own device, with an override under **Configure**. When the fan is
on the firmware's own `auto`, the thermostat tile mirrors the head's actual decoded blower
stage instead of freezing on the token `auto`. Display only — it never changes what I
command. No new entities, and nothing to do if your head doesn't publish a `stage` sensor.
